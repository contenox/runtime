# Contenox Runtime Agentic Benchmark

This benchmark measures the shipped `contenox` CLI in the same shape users run it:
chain execution, chat sessions, local tools, SQLite state, backend resolution, and
`modeld` local inference. `modeld` transport-level probes may be added as sidecars,
but headline numbers must come from `contenox`.

## Goals

- Measure end-to-end turn latency for agentic CLI workflows.
- Measure prompt/completion token usage and per-task durations from `--trace`.
- Compare context windows, backends, models, and platforms with repeatable steps.
- Keep setup portable across Windows, Linux, and macOS.

## Required Artifacts

Record these with every run:

- `contenox version`
- `modeld version --json`
- OS, CPU, GPU/NPU inventory, RAM/VRAM, driver/runtime versions
- accelerator memory capacity, memory bandwidth, memory bus width when published,
  and accelerator interconnect/topology for multi-device runs
- model name, backend type, model path, digest/checksum, quantization
- KV precision, context tier, output-token tier, and backend/runtime profile
- chain file used
- command line, environment variables, and working directory
- raw stdout, raw stderr, modeld logs, and parsed result JSON/CSV

## Setup

Use an isolated workspace and database so runs do not inherit user history.

Unix:

```sh
export BENCH_ROOT="$PWD/.bench/contenox-agentic"
export CONTENOX_DATA_ROOT="$BENCH_ROOT/modeld"
mkdir -p "$BENCH_ROOT/workspace" "$BENCH_ROOT/data" "$BENCH_ROOT/logs"
cd "$BENCH_ROOT/workspace"
contenox --data-dir "$BENCH_ROOT/data" init
```

Windows PowerShell:

```powershell
$env:BENCH_ROOT = "$PWD\.bench\contenox-agentic"
$env:CONTENOX_DATA_ROOT = "$env:BENCH_ROOT\modeld"
New-Item -ItemType Directory -Force "$env:BENCH_ROOT\workspace", "$env:BENCH_ROOT\data", "$env:BENCH_ROOT\logs" | Out-Null
Set-Location "$env:BENCH_ROOT\workspace"
contenox --data-dir "$env:BENCH_ROOT\data" init
```

Register the backend exactly as a user would. Examples:

```sh
contenox backend add llama --type llama --url "$HOME/.contenox/models/llama"
contenox config set default-provider llama
contenox config set default-model <model-name>
contenox config set default-max-tokens 256
contenox config set default-think off
```

```sh
contenox backend add openvino --type openvino --url "$HOME/.contenox/models/openvino"
contenox config set default-provider openvino
contenox config set default-model <model-name>
contenox config set default-max-tokens 256
contenox config set default-think off
```

Start `modeld` separately and keep its stderr/stdout logs. Backend/device selection
is part of the benchmark input. `CONTENOX_DATA_ROOT` must match `modeld
serve --data-root`; the runtime probes that lease root when resolving local
llama/OpenVINO models.

For OpenVINO runs, record whether `CONTENOX_OPENVINO_DEFER_PREFILL` is unset or
enabled. The flag is an experimental no-cold-store fast path; do not mix flagged
and unflagged runs in the same comparison table.

Unix:

```sh
CONTENOX_MODELD_BACKEND=openvino \
CONTENOX_OPENVINO_DEVICE=GPU \
modeld serve --data-root "$BENCH_ROOT/modeld" >"$BENCH_ROOT/logs/modeld.out" 2>"$BENCH_ROOT/logs/modeld.err" &
```

Windows PowerShell:

```powershell
$env:CONTENOX_MODELD_BACKEND = "openvino"
$env:CONTENOX_OPENVINO_DEVICE = "GPU"
Start-Process -FilePath ".\modeld.cmd" `
  -ArgumentList "serve --data-root `"$env:BENCH_ROOT\modeld`"" `
  -RedirectStandardOutput "$env:BENCH_ROOT\logs\modeld.out" `
  -RedirectStandardError "$env:BENCH_ROOT\logs\modeld.err"
```

Sanity checks:

```sh
modeld version --json
modeld status --json
contenox doctor
contenox model list
```

## Workloads

Use the default chains generated by `contenox init` unless a run explicitly names
a benchmark-specific chain. Always save the exact chain file.

### 1. Stateless Pipeline

Mirrors CI/script usage.

```sh
contenox run \
  --db "$BENCH_ROOT/local.db" \
  --data-dir "$BENCH_ROOT/data" \
  --trace --raw \
  --context <ctx> \
  --max-tokens 256 \
  --input @prompts/repo-summary.txt \
  >"$BENCH_ROOT/logs/run-ctx-<ctx>.out" \
  2>"$BENCH_ROOT/logs/run-ctx-<ctx>.err"
```

### 2. Persistent Chat Loop

Mirrors normal agent/chat use. This is the primary context-growth test because
history is stored and resent through the real runtime path.

```sh
contenox --db "$BENCH_ROOT/local.db" --data-dir "$BENCH_ROOT/data" session new bench-ctx-<ctx>

contenox chat \
  --db "$BENCH_ROOT/local.db" \
  --data-dir "$BENCH_ROOT/data" \
  --trace --raw \
  --context <ctx> \
  --max-tokens 256 \
  --input @prompts/turn-01.txt \
  >"$BENCH_ROOT/logs/chat-ctx-<ctx>-turn-01.out" \
  2>"$BENCH_ROOT/logs/chat-ctx-<ctx>-turn-01.err"
```

Repeat for `turn-02.txt` through `turn-N.txt` without deleting the DB.

### 3. Tool-Using Agent Loop

Mirrors coding-agent behavior. Use `--auto` only in a disposable benchmark
workspace.

```sh
contenox chat \
  --db "$BENCH_ROOT/local.db" \
  --data-dir "$BENCH_ROOT/data" \
  --trace --raw --steps \
  --shell --auto \
  --local-exec-allowed-dir . \
  --context <ctx> \
  --max-tokens 256 \
  "Inspect this repository, run the relevant tests, and report the result." \
  >"$BENCH_ROOT/logs/tools-ctx-<ctx>.out" \
  2>"$BENCH_ROOT/logs/tools-ctx-<ctx>.err"
```

## Matrix

Run each workload across:

- context: `4096`, `8192`, `16384`, `32768`, then stop when `modeld` or the model
  clamps/fails or p95 latency exceeds the workload budget
- backend/device: `llama` CPU, `openvino` CPU, `openvino` GPU/iGPU
- hardware topology: single accelerator, tensor-parallel, pipeline-parallel,
  replicated-weights/sharded-KV, or vendor/runtime-specific equivalent when available
- NPU: run as an explicit negative/control case; record whether `modeld` rejects
  it or falls back
- rounds: at least 5 per cell; report p50/p90/p95

## Metrics

Primary metrics come from the outer `contenox` process:

- wall-clock command duration
- time to first output token or nearest available first-output measure
- time per output token or nearest available decode-token measure
- success/failure
- `--trace` task duration
- `--trace` prompt tokens, completion tokens, total tokens
- tool-call count and tool execution duration when tools are enabled
- error text for context overflow, backend mismatch, unsupported feature, timeout

Secondary `modeld` sidecar data:

- selected backend/device from `modeld.err`
- `modeld status --json` before and after workload
- `modeld version --json`
- modeld load/clamp/errors from logs
- prefix-cache hit/miss, snapshot restore cost, residency events, and KV eviction
  telemetry when the backend reports them

Do not headline direct `modeld` transport measurements unless they are clearly
labeled as internal diagnostics. The acceptance question is what a `contenox`
user gets.

## Interpretation

- Autocomplete/editor viability: low end-to-end latency on short-output runs.
- Interactive chat viability: stable p95 latency at `8k` and `16k`.
- Agent-loop viability: tool workload succeeds repeatedly at `16k+` without
  context overflow or runaway latency.
- Long-context viability: `32k+` runs succeed and remain materially usable.
- Hardware-context viability: the context tier passes the latency and quality budget,
  not only the memory-fit check.

If a backend only looks good in direct `modeld` tests but fails or becomes slow
through `contenox`, it does not support the real workflow yet.

Do not report "maximum context" as a capability unless the tier passes the end-to-end
latency and quality gates. A larger VRAM or unified-memory pool is a fit signal, not a
certified interactive-context result.

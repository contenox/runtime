# Support

This project uses GitHub issues as the main place for community support and maintainer coordination. You do not need a polished report. Pick the closest template and share what you know.

## Where To Start

- Bug reports: use `Bug Report` when something broke, regressed, or behaved differently than expected.
- Rough edges: use `Rough Edge` when something is confusing, surprising, awkward, or harder than it should be.
- Ideas: use `Idea or Feature` for new behavior, integrations, workflow improvements, or design proposals.
- Questions: use `Question or Help` when you are trying to understand how something works.
- Docs: use `Docs Improvement` for missing, outdated, or confusing documentation.
- Larger contribution work: use `Contribution Proposal` before investing in a non-trivial PR.

## What Helps

For command-line issues, include the command, output, Contenox version, operating system, provider/backend, and model when relevant.

For chain or tool behavior, include the smallest chain snippet, tool policy, or setup detail that shows the problem.

For ACP/editor behavior, include the client name and version if you know it, plus whether you are using `contenox acp` or `contenox acpx`.

For docs feedback, point to the page, section, command help, or missing topic.

## Security

Do not report vulnerabilities in public issues. Follow [SECURITY.md](SECURITY.md) and email `hello@contenox.com`.

Examples include sandbox escapes, unsafe local tool behavior, credential exposure, auth bypasses, or ways to make Contenox execute actions outside the intended policy.

## Response Expectations

Maintainers will triage when they can. Some issues may be relabeled, merged with related work, converted into follow-up tasks, or closed if they are outside the project scope.

A clear issue is welcome even when nobody can implement it immediately.

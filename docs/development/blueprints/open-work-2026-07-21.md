# Open work — session handoff, 2026-07-21

## Preamble — what contenox is (the maintainer, in their own terms, 2026-07-21)

Contenox is an **editor-runtime-agent**: short-lived, alive until the job is
done — not a serve. A mission is a **subagent** of that process; it reports
back and notifies exactly the parent that fired it. **Beam is like VS Code —
an editor and/or configuration wizard.** The server — the thing that runs
24/7, possibly a whole server fleet — **is modeld**. Getting this wrong is
easy, and daemon-shape accreting around serve is exactly the drift to guard
against.

This is **not** the "hosts your personal assistants" space. It is a product
for **operators, DevOps engineers, and SWEs** — anyone who knows the CLI, can
ssh into something, and can do a quick `contenox serve .` with a port-forward
or a public-net bind (token auth and `serve --remote` are wanted features).
You should not even *need* Beam.

There is a very specific, purpose-made batch of work where this is super
handy, especially in the sysadmin space, and in regular work: prompting the
model to submit a timesheet via remote tools after dumping the day's work
into it; letting it read the git history; quickly reviewing work landed by
teammates or AI agents by cloning into a barely-set-up VM launched on the
spot, so your own environment is never disrupted; running AI agents on
multiple VMs or devices **on the same task, then just reviewing and deciding
what to take** — a practice the classic IDE and classic GitHub workflows
cannot express or even make public; white-room engineering via AI; setups you
can simply drop contenox into.

**Not everything has to be a contenox feature — it just has to close the
gaps.**

## 2. The ontology-correction refit (LANDED 2026-07-21)

The serve-forwarding of `/mission` was a mistake (built on the false premise
"standalone acp has no fleet kernel" — the kernel is an embeddable library).
The refit is now BUILT (uncommitted, as everything is). What landed:

- **DONE — In-process missions in `contenox acp`**: the acp profile (acpx gets
  none of it) embeds, over the one shared db handle and ONE shared
  `libbus.NewSQLite` bus, `agentregistryservice` + publisher-wired
  `missionservice` + `agentinstance` kernel + `operatorinbox` + a started
  `reportrouter` + `fleetservice`, plus `chainagents.Discover` so declared
  `agent-*.json` chains and registered external agents are dispatchable
  (`runtime/contenoxcli/acp_cmd.go`, `buildInProcessFleet`). `/mission`
  dispatches in-process; the unit is a child subprocess; its report is
  delivered LIVE into the firing editor session (`missionReportDeliverer` →
  `acpsvc.Transport.DeliverToContenoxSession`, mapping the firing session's
  contenox id onto the ACP connection). The operator inbox is now only the
  no-live-parent fallback. Kernel/router teardown on shutdown reaps the child
  (mission lifetime ≤ acp process lifetime). Capability always-on — the serve
  reachability probe is gone from `hasMissionCapability`. Zero-setup for
  registry-install users. A dispatched UNIT (chain env set) hosts NO fleet of
  its own — the guard against double-routing on the broadcast SQLite bus.
- **DONE — Forwarding demoted to explicit opt-in**: only when
  `CONTENOX_SERVER_URL` is set does the forwarder replace the in-process
  dispatcher (`acp_mission_forward.go`, now env-only — presence auto-discovery
  and the loopback default REMOVED). Advertisement is unconditional; the
  serve-centric teaching error survives only on this path, at invocation. The
  in-process confirmation now says reports arrive in THIS session; the forwarded
  one still names the operator inbox honestly.
- **DONE — Acceptance**: a house-idiom in-process e2e
  (`runtime/contenoxcli/acp_mission_inproc_e2e_test.go`, `testing.Short()`-gated,
  `-race`, no serve) proves advertise → invoke → live report with
  `contenox.missionReport` `_meta` (NOT the inbox) → durable mission record →
  child reaped on editor shutdown, plus the dead-serve opt-in teaching error.
  The forwarding e2e was refit to the opt-in shape; the acpsvc capability/
  forwarding unit tests updated.
- **STILL OPEN — design note**: a child's permission asks routing to the PARENT
  session's permission UI (Zed's cards) with the unattended answerer as fallback
  — NOT wired (no unattended answerer in the embedded acp kernel; dispatched
  units run bounded/`--auto`). Would remove the `default-mission-policy` friction.
- **STILL OPEN**: `mission fire --wait` hosting the kernel in-process (the CLI
  blocking form) — not built in this slice.

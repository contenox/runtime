## 2. The ontology-correction refit (decided direction, NOT built)

The serve-forwarding of `/mission` was a mistake (built on the false premise
"standalone acp has no fleet kernel" — the kernel is an embeddable library).
The refit, agreed in direction:

- **In-process missions in `contenox acp`**: embed kernel + missionservice +
  reportrouter (libraries over the local.db the process already opens).
  `/mission` dispatches in-process; the unit is a child subprocess of the acp
  process; reports `DeliverToSession` into the firing session — live in the
  editor (this FIXES the parent-gone fallback the forwarding slice
  documented). Mission lifetime = the acp process ("alive until I kill it or
  the mission ends"). Capability always-on — no serve probe. Registry-install
  (Zed/ACP registry) users get the whole journey zero-setup.
- Forwarding machinery: demote to explicit opt-in or drop at review (the
  discovery/probe code is reusable). Rewrite the confirmation/teaching texts
  that currently teach serve as the center.
- Design note worth taking: a child's permission asks could route to the
  PARENT session's permission UI (Zed's cards) with the unattended answerer
  as fallback — would remove the `default-mission-policy` config friction.
- `mission fire --wait` can host the kernel in-process (the CLI blocks for
  the job's lifetime) — serve becomes unnecessary for the blocking form;
  detached fire may still use a serve where one exists.
